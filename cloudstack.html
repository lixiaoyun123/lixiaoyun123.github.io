Cloudstack数据转移

一、 更改二级存储

更改二级存储之前，需要强调：每个二级存储都需要安装系统VM。

首先，在UI界面禁用资源域

然后，管理节点停止cloudstack-management服务 

# systemctl stop cloudstack-management

接下来，选择地址新建一个二级存储：

如果新的地址未安装NFS，需要先进行安装 

# yum install nfs-utils

在存储服务器上，为二级存储创建 NFS 共享，例如： 

# mkdir -p /export/secondary2

要将新目录配置为 NFS 导出，请编辑 /etc/exports。使用 rw、async、norootsquash、nosubtreecheck 导出 NFS 共享。例如： 

# vi /etc/exports

插入以下行（对应上面新建的文件路径）： 

/export *(rw,async,no_root_squash,no_subtree_check)

导出 /export 目录 

# exportfs -a

如果二级存储与管理节点不在同一服务器上，则需要在管理服务器上，为二级存储创建一个挂载点。例如： 

# mkdir -p /mnt/secondary

在您的管理服务器上装载辅助存储。将下面的示例 NFS 服务器名称和 NFS 共享路径替换为您自己的。 

# mount -t nfs nfsservername:/export/secondary2 /mnt/secondary

编辑 /etc/sysconfig/nfs 文件 

# vi /etc/sysconfig/nfs

取消注释以下几行（或直接添加在尾部）： 

LOCKD_TCPPORT=32803
LOCKD_UDPPORT=32769
MOUNTD_PORT=892
RQUOTAD_PORT=875
STATD_PORT=662
STATD_OUTGOING_PORT=2020

编辑 /etc/sysconfig/iptables 文件。

# vi /etc/sysconfig/iptables

在 INPUT 链的开头添加以下几行，其中 是您将使用的网络： 

-A INPUT -s <NETWORK> -m state --state NEW -p udp --dport 111 -j ACCEPT
-A INPUT -s <NETWORK> -m state --state NEW -p tcp --dport 111 -j ACCEPT
-A INPUT -s <NETWORK> -m state --state NEW -p tcp --dport 2049 -j ACCEPT
-A INPUT -s <NETWORK> -m state --state NEW -p tcp --dport 32803 -j ACCEPT
-A INPUT -s <NETWORK> -m state --state NEW -p udp --dport 32769 -j ACCEPT
-A INPUT -s <NETWORK> -m state --state NEW -p tcp --dport 892 -j ACCEPT
-A INPUT -s <NETWORK> -m state --state NEW -p udp --dport 892 -j ACCEPT
-A INPUT -s <NETWORK> -m state --state NEW -p tcp --dport 875 -j ACCEPT
-A INPUT -s <NETWORK> -m state --state NEW -p udp --dport 875 -j ACCEPT
-A INPUT -s <NETWORK> -m state --state NEW -p tcp --dport 662 -j ACCEPT
-A INPUT -s <NETWORK> -m state --state NEW -p udp --dport 662 -j ACCEPT

运行以下命令： 

# service iptables restart
# service iptables save

如果客户端和服务器之间使用 NFS v4 通信，请将您的域添加到管理程序主机和管理服务器上的 /etc/idmapd.conf 

# vi /etc/idmapd.conf

删除 idmapd.conf 中 Domain 行开头的字符 # 并将文件中的值替换为您自己的域。在下面的示例中，域是 cloud.priv 

Domain = cloud.priv

确保 NFS 和 rpcbind 正在运行 

# service rpcbind start
# service nfs start
# chkconfig nfs on
# chkconfig rpcbind on

将原二级存储内所有内容复制到新的二级存储中。。。尽量这么做，不然模板ISO，什么的全没了。 

# cp -r /export/secondary/* /export/secondary2/

修改数据库，主要是cloud库的image_store表

# mysql -u root -p

sql> use cloud;
sql> select * from image_store;

修改url为新的二级存储地址,例如： 

update image_store set url='nfs://172.18.197.3/export/secondary2' where id = 1;

如果使用单独的 NFS 服务器，请执行此步骤。如果将管理服务器用作 NFS 服务器，则不得执行此步骤（即卸载前面挂载的目录）。

# umount /mnt/secondary
# rmdir /mnt/secondary

重启cloudstack-management 

# systemctl restart cloudstack-management

启动资源域

二、 更改主存储

在ui上新建主存储，需要确保新的节点已安装nfs，且新的路径需要在/etc/exports配置共享。例如：

# vi /etc/exports

添加（/data为新的主存储所在路径）：

/data *(rw,async,no_root_squash,no_subtree_check)

正常添加一个新的主存储

然后将原主存储启用维护模式。时间较长，耐心等待。这个过程会停掉所有使用该存储的虚拟机

也有比较麻烦的一步，需要一个一个将实例存储迁移到新增的上面

全部迁移完毕。。。然后就可以删除该主存储了。。可以强制删除。不用担心虚拟机无法启动，再次启动虚拟机时，它会在新的主存储启动。

可以启动虚拟机

相关卷在第二个主存储上

三、 转移管理节点

现有一个管理节点，一个计算节点，需将管理节点转移至新的服务器上。

首先，确定如果新的管理节点和数据库不在同一个主机，需要设置数据库配置文件

# vi /etv/my.cnf

[mysqld]修改为以下内容：

[mysqld]
innodb_rollback_on_timeout=1
innodb_lock_wait_timeout=600
max_connections=1050
log-bin=mysql-bin
binlog-format = 'ROW'
bind-address = 0.0.0.0

和之前的相比多了一行 bind-address = 0.0.0.0。因为数据库要允许其它主机连接。max_connections值为350*管理节点的个数。

# systemctl stop cloudstack-management

创建新的管理节点进行基本配置。

安装管理程序

# yum -y install cloudstack-management

连接数据库初始化管理程序

# cloudstack-setup-databases cloud:password@172.18.197.1
# cloudstack-setup-management

使用新管理节点IP访问WEBUI修改全局变量 host 为新节点ip

由于host 并非原管理节点修改，计算节点主机配置配置文件需要手动修改

# vi /etc/cloudstack/agent/agent.properties

将 host 修改为新管理节点IP

重启cloudstack-agent 

# systemctl restart cloudstack-management

新的管理节点不能与原系统VM通信，需要删除掉原有系统VM。会自动生成新的VM。

等一段时间，便可看到新的系统VM

等待系统VM生成，状态变为running 和 up 管理节点即可正常使用

四、 数据库节点转移

数据库意外失效，且有备份数据时，更换cloudstack使用的数据库

cloudstack与数据库建立连接使用的是 cloud 用户，操作的两个库是cloud和cloud_usage。

首先 停止 cloudstack-management

# systemctl stop cloudstack-management

备份一份数据出来，包括 cloud cloud_usage两个库的所有内容

# mysqldump -u root -p --databases cloud cloud_usage > cloud.sql

备份完毕，关闭数据库，假装数据库意外损坏

# systemctl stop mariadb

新数据库节点操作：编辑数据库配置文件

# vi /etc/my.cnf

修改：

innodb_rollback_on_timeout=1
innodb_lock_wait_timeout=600
max_connections=700
log-bin=mysql-bin
binlog-format = 'ROW'
bind-address = 0.0.0.0

重启数据库

# service mysqld restart

进入数据库控制台

# mysql -u root -p

创建cloud用户并 赋予 权限

grant process on *.* to 'cloud'@'localhost' identified by 'password';
grant all privileges on *.* to 'cloud'@'localhost';
grant process on *.* to 'cloud'@'%' identified by 'password';
grant all privileges on *.* to 'cloud'@'%';

刷新权限 退出控制台

flush privileges;
exit

将之前备份的sql文件复制到本节点(/root/下)，将其中数据备份到本节点数据库 

mysql -u root -p < cloud.sql

现在数据已经转移到新的数据节点了，重点是 管理节点设置与新数据库管理节点建立连接 

# cloudstack-setup-databases cloud:password@172.18.197.5
# cloudstack-setup-management
